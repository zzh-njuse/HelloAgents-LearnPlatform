# Stage 4 Slice 4 OCR 独立审查记录

日期：2026-07-23

## 审查背景

本次审查覆盖 Stage 4 Slice 4 的 MCP 增量实现，包括代码执行 MCP、Wolfram 科学工具准入、课程/练习/Tutor 工具授权链路、能力探测、队列、数据模型、Web 集成和测试。审查发生在人工 smoke 之后；Java/C++ 及练习生成的系统性质量问题已明确作为 Slice 5 输入，不在本次 MCP 收尾中扩项修复。

## 范围与执行方式

- 使用脱敏普通源码副本，主动排除 `.env`、API key、私有连接地址、内部域名、上传原文、敏感 prompt、日志、绝对路径和 provider 配置。
- 共扫描 25 个风险分块、50 个文件副本；所有分块均使用 `--audience human --concurrency 1 --timeout 10`。
- 25 份报告全部包含完整 Summary，没有零文件扫描或预算截断。
- OCR 共返回 267 条原始 comments，约消耗 4,326,733 tokens。原始输出保存在 `reviews/raw/ocr-slice4-final-*.txt`。
- OCR comments 含跨块重复、脱离 Spec/ADR 的泛化建议、测试源码静态推断和既有 Slice 行为，因此不能按数量直接等同于有效缺陷。

## 采纳并修复

### High

1. **代码执行客户端未强制校验 canonical schema hash**
   - MCP 握手虽然计算了 input/output schema hash，但此前未与共享合同常量比较。
   - 现已在任何 `call_tool` 之前比较两侧 hash；不一致稳定返回 schema drift，不执行远端 Tool。

2. **Code Run 幂等创建存在并发竞态**
   - 同一 workspace/idempotency key 并发插入可能直接暴露数据库唯一约束错误。
   - 现已捕获唯一约束冲突，回读权威 Job；相同 request hash 返回既有 Run，不同 hash 稳定返回 409。

3. **Workspace MCP policy 首次创建存在并发竞态**
   - 现已在唯一约束冲突后回滚并锁定权威 policy，再应用本次更新。

### 高置信 Medium

1. `ExecutionAdapter` 的延迟 HTTP client 初始化/关闭加入锁，避免 worker 并发创建或关闭同一 client。
2. Judge0 非 JSON/非对象响应现在归类为 `invalid_tool_result`；其他传输错误归类为后端不可用，不回显远端正文。
3. MCP `/readyz` 现在只接受后端 2xx，不再把任意 HTTP 响应视为 ready。
4. MCP lifespan shutdown 在 startup 未完成时不再引用未绑定变量。
5. `RunCodeOutput` 对 stdout、stderr、compile output 和 runtime 增加合同级长度上限。
6. Fake execution backend 的 output-limit 状态修正为 Judge0 status 8，与产品映射一致。
7. 科学题 suitability 在缺失 objective key 时稳定判为 unsupported，不再索引空列表。
8. Tutor 问题统一 trim 并拒绝纯空白；可选 `code_run_id` 同样拒绝空白。
9. enqueue 失败时 CodeLab Job 与 Run 同步进入 `queue_failed`，避免界面永久显示 queued。
10. MCP server 对基础设施失败记录稳定类别日志，但不记录 URL、原始异常或私有正文。

## 排除或暂缓

- **Slice 5 输入**：Java/C++ 练习持续生成失败、编程/科学练习的系统性 artifact 质量、跨练习去重与生成预算调优。它们需要单独 Spec/ADR 复核，不能借 OCR 收尾局部硬编码。
- **鉴权泛化建议**：当前产品仍按既有 workspace 单用户/self-host 边界运行；不在 Slice 4 临时引入新的身份系统。
- **动态 MCP 市场、任意 Tool discovery、更多 capability**：违反已接受的 Spec 004 / ADR 006 范围。
- **让 Tool observation 自动写入学习事实**：明确违反事实边界。
- **删除链孤儿、Tutor session 级联缺失等报告**：复核 ORM 外键和显式清理后不成立。
- **大量索引、命名、格式和测试实现风格建议**：未证明用户可见错误或数据风险，本轮不盲改。
- **要求记录完整异常正文**：与仓库脱敏日志规则冲突；只采纳稳定错误类别日志。
