# Stage 4 Slice 3 教学 Skill 运作示例

状态：Spec 003 的说明性示例，不替代结构化合同或 eval

日期：2026-07-18

## 1. 场景

用户正在学习“软件管理”课程，已经读过“大教堂模式与市集模式”的课节，并完成过两套练习。

用户在 Tutor 的“整门课程”范围选择“诊断式辅导”，提问：

> 我接下来该学什么？

### 当前产品事实

课程证据：

- `e1`：大教堂模式强调集中设计、较长发布周期和有限参与者。
- `e2`：市集模式强调早发布、常发布和广泛用户反馈。
- `e3`：选择模式需要考虑项目成熟度、参与者规模、沟通机制和反馈速度。

已授权学习状态：

- confirmed weakness：`能分析实际项目适合哪种开发模式`，由两次独立错误支持。
- provisional suggestion：`掌握两种模式的基本差异`，只有一次负向证据。
- active Memory：`我需要继续巩固：能分析实际项目适合哪种开发模式`。
- Lesson Completion：用户已经完成“大教堂模式与市集模式”课节阅读。

这里没有把用户答案、rubric、AI feedback、projection score 或 evidence 正文发送给 Skill。

## 2. 离线基线 Tutor 可能产生的结果

Stage 3 基线 Tutor 只受到通用“回答并引用”的约束，可能给出。该基线只用于离线配对 eval，不是 Slice 3 用户可以选择的模式：

> 根据学习记录，你需要继续巩固两种开发模式的区别，以及分析实际项目适合哪种开发模式。建议重新阅读《大教堂与市集》，并继续做练习。

这段话不一定错误，但存在三个问题：

- 基本是 Memory 的复述，没有解释为何先学这个。
- 没区分 confirmed weakness、provisional suggestion 和已经完成阅读。
- “继续做练习”不是足够具体的学习动作。

## 3. Skill 的内部计划

第一 provider 调用必须返回结构化计划，例如：

```json
{
  "intent": "study_planning",
  "queries": [
    "大教堂模式 市集模式 适用场景 判断标准",
    "项目特征 发布周期 参与者规模 反馈速度"
  ],
  "learning_context_use": "required",
  "teaching_moves": ["focus", "explain", "next_action"]
}
```

这里的 `study_planning` 不是由“接下来学什么”关键词硬编码得到。等价表达如“下一步怎么安排”“现在最应该补哪块”应走相同策略；反例“市集模式是谁提出的”则应将学习状态判为 `irrelevant`。

## 4. 确定性上下文选择

Product service 在模型回答前执行：

1. 验证 Skill 为 allowlist 中的 `evidence-guided-diagnostic-scaffold` v1，hash 与部署定义一致。
2. 只保留当前 Workspace/Course 范围内的状态。
3. 把 confirmed weakness 排在 provisional suggestion 前。
4. 把 Completion 标记为“读过”，不得转译为“已掌握”。
5. 使用计划中的两个 query 检索当前 Course Version，建立 `e1-e3` citation ledger。
6. 记录安全 trace：Skill version、选择了 1 条 confirmed/1 条 provisional/1 条 completion、2 次检索；不记录正文。

## 5. Skill 最终回答示例

用户看到的回答可以是：

### 先练“根据项目条件选择开发模式”

你已经完成了两种模式基础内容的阅读，但现有两次独立练习证据都指向同一个缺口：能够描述两种模式，还不能稳定地把项目条件映射到模式选择。因此下一步不必从头重读整课，应优先练习“为什么这个项目更适合某种模式”。

大教堂模式通常对应更集中的设计和较长的发布节奏；市集模式依赖更早、更频繁的发布以及更广泛的反馈。[1][2] 真正决定选择的不是项目名称，而是成熟度、参与者规模、沟通成本和获得反馈的速度。[3]

### 下一步

回看课节中关于发布节奏和参与者协作的部分，然后为下面两个场景各写一句选择与理由：

1. 需求稳定、参与者很少、发布前需要严格统一设计的内部系统。
2. 已有可运行原型、用户很多、可以频繁收集反馈的开源工具。

答完后再生成一套当前课节练习，重点验证“选择依据”，而不是重复背诵两种模式的定义。

引用：

1. 《大教堂与市集》> 模式定义 > 第 14-20 页
2. 《大教堂与市集》> 早发布与用户反馈 > 第 21-27 页
3. 《软件管理》> 项目条件与协作 > 第 33-38 页

## 6. 为什么这不是固定答案

若产品事实变化，Skill 的行为也会变化：

- 只有一次错误时，只能说“有一个初步信号”，不能断言薄弱点。
- 用户尚未完成课节时，可以建议先读对应部分，不能说“无需重读”。
- 已有后续正向证据且 Weakness resolved 时，不应继续把它列为首要缺口。
- Memory policy 关闭时，回答只能基于课程证据给一般学习路径，不能声称知道用户的薄弱点。
- 没有课程 citation 时，可以描述安全的学习状态，但不能讲解两种模式的课程事实。

## 7. 不应触发个性化的反例

用户问：

> 《大教堂与市集》首次发表于哪一年？

计划应接近：

```json
{
  "intent": "concept_explanation",
  "queries": ["大教堂与市集 首次发表 年份"],
  "learning_context_use": "irrelevant",
  "teaching_moves": ["explain"]
}
```

回答应直接给出课程资料支持的年份和引用，不附加“你的薄弱点是……”或强制安排练习。这是防止 Skill 退化成“每次都套 Memory”的关键反例。

## 8. 多轮中的下一步

如果用户回答了上面的两个场景，下一 Turn 可以选择 `probe` 或 `check`：指出理由中缺失的判断维度，再追问一个更窄的问题。它仍然不能直接写回 Mastery 或 Memory；只有正式 Practice/Attempt 或既有 Learning Event 合同才能改变学习状态。
