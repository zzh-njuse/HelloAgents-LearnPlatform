# Stage 4 过程复盘

状态：持续维护中的阶段性记录，不是 Stage 4 最终复盘
首次记录：2026-07-24

## 1. 文档用途

本文用于在 Stage 4 收尾过程中及时记录已经出现的协作和工程信号，避免等到项目结束后只凭印象复盘。

当前结论均为阶段性判断。项目完成后的总复盘仍需结合后续 Stage、实际使用结果、提交历史、验证记录和最终维护成本重新评估，不在本文提前给个人、模型或协作方式下最终结论。

## 2. 当前可以直接确认的问题

### 2.1 后期 Slice 过大

Stage 4 后期 Slice 同时包含了过多不同性质的风险：

- Slice 3 同时涉及版本化教学 Skill、Tutor 编排、检索与学习状态综合、异步任务、历史隔离和前端投影。
- Slice 4 同时涉及代码执行、Java/C++/Python canonical contract、Judge0、MCP、Wolfram、授权预算、科学验证、公式渲染和编程交互。
- Slice 5 虽名义上只是稳定化，却横跨生成、schema、引用、去重、repair、预算、编译执行、科学验证、评分权威、迁移和 Web smoke 修正。

这些 Slice 不只是“代码量较大”，而是同时叠加了多个独立风险轴。任务包即使覆盖全面，执行模型也需要一次保持过多跨模块不变量，导致局部实现看似符合要求、完整链路仍然失败。

目前可以将“后期 Slice 过大、风险边界拆分不足”视为 Stage 4 进展明显变慢的直接原因之一。

### 2.2 测试与真实流程脱节

Stage 4 已有大量单元测试和 focused tests，但后期多次由真实环境或人工 smoke 才发现关键问题，包括：

- 编译器在 PATH 中不等于完整工具链健康。
- Docker test 镜像缺少 Java/C++ compiler，无法承担三语言稳定 Gate。
- Java starter/canonical wrapper 在真实执行时出现文件名与 public class 问题。
- 生成预算在结构修复与 specialized repair 组合下耗尽。
- 试运行结果跨题目残留。
- 专注模式和 Tutor 控件在真实页面中的状态与布局异常。
- 重复检测可以拒绝结果，但生成阶段未充分避免重复。

Correction 过程中还出现过以下不能证明产品行为的测试形式：

- 只检查源码字符串。
- 复制一遍产品内部的 `if/raise` 逻辑。
- 用整数相加代替真实预算编排。
- 调用了 helper，却没有执行真实 `execute_generation`、worker、数据库提交或浏览器路径。

因此，当前可以确认的第二个直接问题是：

> 测试数量不少，但测试金字塔失衡；单元和局部行为测试较多，跨 Provider、编排、数据库、外部工具和 Web 的系统测试不足，且真实 smoke 进入得偏晚。

“测试通过”与“用户主流程可靠”在 Stage 4 后期没有保持足够强的对应关系。

## 3. 当前尚不下结论的问题

### 3.1 用户与 Codex 的交互方式

当前不能简单认定用户审核不足、确认过快或交互方式本身有问题。

本项目规模和技术跨度已经使普通人工 Gate 难以直接判断以下细节：

- schema 与 Pydantic 验证是否真正一致；
- repair 是否越过不可变字段权限；
- 测试是否实际经过生产路径；
- 编译、容器和远程工具的环境结论是否可重复；
- 错误码、预算和异步状态是否在完整调用链中成立。

这些问题本应主要由可复现证据、自动化验证、Codex 独立复核和外部 review 承担，而不是要求用户逐项判断实现细节的真实性。

但用户 Gate 应如何进一步简化、哪些决策必须保留人工判断、Codex 应以什么形式呈现证据，目前仍需观察。该问题留到项目级复盘，不在 Stage 4 过程记录中仓促归因。

### 3.2 GLM 实现能力与交接方式

GLM 多轮实现中确实出现了局部理解、假测试、错误合同沿用和完成声明过强等问题；同时，任务包本身过大、真实环境证据不足和验收路径定义偏晚也提高了实现难度。

因此目前只记录现象，不把 Stage 4 的困难单独归因于 GLM，也不提前认定改由 Codex 实现就一定能够一次完成。最终需要比较：

- 不同复杂度任务的 correction 轮数；
- 任务包规模与遗漏率；
- Codex 独立复核发现问题的阶段；
- 实现、交接和复验总成本；
- 最终代码的缺陷率与维护成本。

## 4. 后续阶段的候选改进

以下是待在 Stage 4 最终总结或项目级复盘中确认的候选措施：

1. 一个 Slice 默认只承担一个主要风险轴；外部工具接入、生成合同稳定化和复杂 Web 交互应优先拆开。
2. Spec 接受后，先建立少量真实编排或端到端验收路径，再开始大范围产品实现。
3. 第一条纵向主路径可运行后立即做最小真实 smoke，不等整个 Slice 完成。
4. 明确区分单元测试、组件测试、编排测试、环境 Gate 和浏览器 smoke，不再只汇报合并后的 passed 总数。
5. 对关键测试增加反事实检查：删除对应产品逻辑后，测试必须失败。
6. GLM correction 连续达到约两轮时暂停，复核是否应重新拆包、补事实盘点或改变验证方式。
7. 概率性 Provider 功能维护固定小样本 eval、失败分类和成功率基线，而不是用单次成功替代稳定性证据。
8. 依赖编译器、容器或远程服务的功能，在主体实现前建立可重复的环境 preflight 与稳定 Gate。
9. 人工 Gate 优先呈现用户行为、失败策略、成本、风险和未验证事项；底层技术真实性由工程证据负责。

这些措施目前是改进候选，不代表已经修改仓库级 Playbook。是否固化为后续 Stage 的强制流程，应在 Stage 4 收尾和项目级复盘中另行决定。

## 5. 最终复盘需要补充的证据

Stage 4 正式关闭时，至少补充：

- 各 Slice 的实际周期、实现轮次和 correction 轮次。
- 问题首次出现、首次被测试发现和首次被真实 smoke 发现的阶段。
- 单元、编排、环境和浏览器验证的覆盖差异。
- 返工主要来自需求变化、合同遗漏、实现偏差、环境差异还是测试失真。
- 文档和任务包规模与实现成功率之间的关系。
- Codex、GLM、OCR 与用户 Gate 分别发现了哪些类型的问题。
- 哪些流程调整应立即进入 Playbook，哪些留待项目完成后的总复盘。

## 6. 当前阶段性判断

截至 2026-07-24，可以先确认两个最直接的问题：

1. Stage 4 后期 Slice 过大，同时承载了过多跨模块风险。
2. 测试与实际用户流程存在脱节，单元和局部测试多，系统级验证不足且进入偏晚。

关于用户交互、文档审核负担、Codex 与 GLM 的职责分配，目前只记录问题，不作最终判断。
