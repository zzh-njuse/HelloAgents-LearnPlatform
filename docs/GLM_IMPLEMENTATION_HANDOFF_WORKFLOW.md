# GLM 实现交接工作流

状态：仓库级执行经验；2026-07-16 根据 Platform Stage 3 Slice 3 的成功交接归档

## 目标与分工

后续非平凡产品编码默认采用“Codex 定义和验收，GLM 实现”的协作方式：

| 参与方 | 主要责任 |
|---|---|
| 人工 | 产品取舍、Spec/ADR Gate、敏感外发与付费操作确认、UI smoke、提交与 push 时点 |
| Codex | 事实盘点、需求澄清、Spec/ADR、任务拆分、GLM 任务包、独立 self-review、OCR 组织、验收与阶段文档 |
| GLM | 在已接受合同内完成正式编码、测试和实现报告，不自行改变产品范围 |

这是开发执行分工，不是产品内的多 Agent 架构，也不构成引入自主多 Agent 的依据。

## 标准流程

```text
Codex 读取指导文档与代码事实
  -> 与人工对齐需求、Spec、ADR 和 Gate
  -> Codex 生成 GLM 实现任务包与接手 prompt
  -> 人工将任务交给 GLM
  -> GLM 实现并运行任务包指定检查
  -> Codex 核对 git diff、合同、测试和敏感边界
  -> 按批准范围执行独立 OCR
  -> 归类并关闭 High / 高置信 Medium
  -> 完整自动化复验与人工 UI Gate
  -> Codex 写总结、下一阶段输入，按人工指令提交与 push
```

任何一环失败都回到最近的权威合同或实现步骤，不用后续 review 掩盖未通过的 Spec/ADR Gate。

## Codex 交付给 GLM 的任务包

每个实现任务包至少包含：

- 仓库路径、当前分支、已知 dirty files 和不可回滚说明；
- 必须完整读取的根指导文档、当前 Stage README、Spec、ADR、review 与输入文档；
- 已获人工接受的 Goal、用户路径、失败行为、完成标准和明确不做项；
- 涉及新页面或重大交互时，已获人工接受的信息架构、页面概念和状态矩阵；GLM 不自由重做导航或布局；
- 精确的实现范围、建议顺序、允许触及的模块和依赖方向；
- API、schema、migration、任务状态、幂等、删除、预算和敏感信息合同；
- 必须新增或更新的测试，以及完整验证命令；
- 禁止事项，例如扩大到下一 Slice、读取或输出 key、回滚未知改动、提交或 push；
- 完成报告格式：修改文件、关键决策、验证结果、未解决问题和需要 Codex 复核的风险。

任务包应足以让 GLM 直接实现，但不能让任务包代替已经接受的 Spec/ADR。若代码事实与任务包冲突，GLM 应停止对应部分并报告，不得自行重写产品合同。

## GLM 实现规则

- 先读取任务包列出的全部文档和相邻代码、测试，再开始修改。
- 保持 `apps -> academic_companion -> hello_agents` 依赖方向，优先沿用仓库模式。
- 只实现当前 Slice，不主动加入 Tutor、Memory、Skill、MCP、多 Agent、认证、计费或其他未批准能力。
- 不读取、输出或提交 API key、`.env`、私有连接地址、内部域名、上传原文、敏感 prompt、日志、绝对路径或 provider 配置。
- 不回滚未知 dirty files，不执行破坏性 Git 命令，不自行 commit 或 push。
- 改变公开行为时同步测试和文档；无法运行的检查必须报告真实原因，不能写成通过。

## Codex 接回后的验收

GLM 的“完成”只表示实现候选已经交回，不等于 Slice Gate 通过。Codex 必须重新：

1. 检查 `git status`、完整 diff 和依赖边界。
2. 对照 Spec/ADR 和任务包核验每个公开合同，排除无意扩展。
3. 独立运行 focused tests、完整回归、Web build、Compose/readiness 和适用 smoke。
4. 检查敏感字段、删除、幂等、重试、取消、晚到结果和 workspace 隔离。
5. 按 Playbook 的隐私和分块规则组织 OCR，结合上下文排除隔离扫描误报。
6. 修复或回交 High 和高置信 Medium：窄小、明确且不改变合同的修复可由 Codex关闭；跨模块或改变合同的修复重新形成 GLM 修正包。
7. 只有自动化、OCR 和人工 Gate 都真实完成后，才写阶段总结并请求提交/push。

## Stage 3 Slice 3 的有效经验

本次交接成功的关键因素：

- Spec 005 / ADR 007 先完成人工 Gate，GLM 没有承担产品决策。
- 任务包把 eval、脱敏运行摘要、Web 入口、测试和明确不做项拆成有序批次。
- GLM 完成主体实现后，Codex 独立运行完整 API、eval、Web 和 Docker 验证，没有复用实现者的“已通过”结论。
- OCR 使用仓库外白名单副本，普通 API/schema/worker/tests 可审，敏感配置和原始资料主动排除。
- OCR 的孤立上下文误报由 Codex 结合 Spec/ADR、完整测试和实际代码排除；真正问题集中修复后再统一复验。
- 人工最终验证运行记录身份、筛选、删除后脱敏和 token 未报告语义，自动化没有冒充 UI Gate。

需要继续避免：

- 在合同未确认时让 GLM 边写边决定 schema、删除或成本语义；
- 用一个过大的任务包同时跨越多个 Slice；
- 只看 GLM 总结、不检查实际 diff；
- 因 OCR 输出多就盲目修改所有评论；
- 让实现者接触无关 provider 配置或真实用户资料；
- 为节省交接成本而跳过 Codex 独立复验和人工 Gate。

## 适用范围

- 默认适用于 Stage 4 及其后的非平凡产品实现、跨模块修复和较大测试建设。
- 需求讨论、事实盘点、Spec/ADR、任务包、review 结论和阶段总结继续由 Codex 处理。
- 文档索引、错别字和不改变行为的极小维护可以由 Codex 直接处理；不得借“小修”绕过需要 GLM 任务包的业务实现。

## Frontend 设计 Gate

新页面或显著改变信息架构的 Slice，在生成 GLM 任务包前必须经过：用户路径 -> 信息架构 -> 页面概念图 -> 状态/响应式矩阵 -> 人工接受。实现者负责忠实落地和发现技术冲突，不负责在编码阶段替用户重新做产品设计。人工 smoke 发现的设计问题仍回写概念和测试，不能把概念图当作免验收依据。
